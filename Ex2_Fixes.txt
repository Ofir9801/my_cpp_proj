To ensure our project complies with the guidelines provided by our tutor, we are submitting fixes for Ex2:


as we want to make sure our project is according the guidlines that given us by our tutor we submit fix for Ex2.

1.incorrect screen filename format:
	We changed the room names to adv-worldXX in accordance with the guidelines.
	Note: Special rooms, such as the menu, remain unchanged.

2.there's apparently no way to configure darkness of room (for using torch) in screen files:
	We added the ability to mark a room as "dark" by adding the letter 'D' to the game file. Room 3 is provided as an example of a dark room.

3.as stated in your instructions, switches' and keys' connections to doors are randomized, which does not allow for their configuration in screen files:
	We reviewed the guidelines and found no requirement stating that the connection between doors, keys, and switches must be manually configured by the room designer. 
	Therefore, we kept the code as is. Nevertheless, we believe that randomizing these connections makes the game harder and more unpredictable for the user.

4. riddles which appear on screen are picked from riddle file by order of appearance (a limiting implementation choice):
	We decided not to change this implementation. We believe this order is better for the room designer, who only needs to match the number of riddles to the number of question marks in the room,	
	without calculating the exact position of each '?' sign.

5.program crashes when no screen files are present in the working directory:
	We fixed this bug. Now, if no screen files are found in the dedicated folder, the program alerts the user and provides an opportunity to fix the issue and retry without rebuilding the project from scratch.

6. springs do not support lateral movement; () change to STAY key (documented):
	Springs now support lateral movement, as demonstrated in Room 2.

7.obstacle of size 2 or more cannot be moved by using combined strength of two players:
	We fixed this issue. The game now calculates the combined force when both players push together.

8. Need to support any number of rooms:
	we modified our code structure to support the dynamic addition of screen files.

9. screen file identifiers are hardcoded - game should support adding additional screen files or changing names of existing files without editing code (if name of "Room01.screen" is changed game does not start):
	This is addressed by the fix mentioned above regarding the dynamic addition of rooms.

10.use named constants / enum instead of literals (Game::showMenu, Bomb::destroyCell etc.):
	We implemented more enum classes to eliminate "magic numbers" and "magic characters."
